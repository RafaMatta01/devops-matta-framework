Pilares do framework

🏗 Infraestrutura como Código

Terraform provisiona toda a infra AWS de forma declarativa e versionada no Git.

🚀 Pipeline CI/CD

Build, teste e deploy automatizados via GitHub Actions do commit ao EKS.

🔐 Segurança & IAM

Least privilege, IRSA e OIDC — zero credenciais hardcoded nos pods.

☸ Orquestração

Kubernetes no EKS com Ingress, namespaces e service accounts por workload.

2 Configurar credenciais AWS (IAM)
⚠ Importante: Se estiver usando AWS Cloud9 ou ambientes com credenciais gerenciadas, desative as credenciais temporárias antes de exportar as variáveis abaixo — caso contrário o eksctl não funcionará corretamente.

Criar usuário IAM devops-matta-user

AdministratorAccess policy + Access Key para autenticação via CLI

$ aws iam create-user --user-name devops-matta-user
$ aws iam attach-user-policy \
  --user-name devops-matta-user \
  --policy-arn arn:aws:iam::aws:policy/AdministratorAccess
# Exportar credenciais no ambiente
$ export AWS_ACCESS_KEY_ID=AKIAIOSFODNN7EXAMPLE
$ export AWS_SECRET_ACCESS_KEY=wJalrXUtnFEMI/K7MDENG
$ aws sts get-caller-identity

Terraform — provisionar S3 + DynamoDB

Infraestrutura de dados para cada estado via IaC — versionada no Git

$ cd app-devops-matta/infrastructure/terraform
$ terraform init
$ terraform plan
$ terraform apply
# Outputs: s3_bucket_name (norte e sul), dynamodb_table_name (norte e sul)
# Anote os nomes — serão usados nos YAMLs
3 Criar e conectar ao cluster EKS
⏱ Atenção: O comando eksctl create cluster pode levar entre 15 e 20 minutos para concluir. Aguarde a finalização antes de prosseguir.

Criar o cluster EKS

Node group com instâncias t3.medium na us-east-1

$ eksctl create cluster \
  --name devops-matta-cluster \
  --region us-east-1 \
  --nodegroup-name standard-workers \
  --node-type t3.medium \
  --nodes 1\
  --node-private-networking \
  --vpc-nat-mode HighlyAvailable

Conectar kubectl ao cluster

Atualiza o kubeconfig e verifica a conectividade

$ aws eks update-kubeconfig \
  --name devops-matta-cluster
$ kubectl get nodes
NAME STATUS ROLES AGE
ip-192-168-x Ready <none> 2m
$ kubectl get svc
4 IAM OIDC Provider + AWS Load Balancer Controller
⚠ Atenção: Substitua <ACCOUNT_ID> pelo ID real da sua conta AWS. Obtenha com: aws sts get-caller-identity --query Account --output text

Associar OIDC Provider + criar IAM Policy

Permite que service accounts assumam roles IAM via IRSA

$ eksctl utils associate-iam-oidc-provider \
  --cluster devops-matta-cluster \
  --approve
$ curl -O https://raw.githubusercontent.com/kubernetes-sigs/aws-load-balancer-controller/refs/heads/main/docs/install/iam_policy.json
$ aws iam create-policy \
  --policy-name AWSLoadBalancerControllerIAMPolicy \
  --policy-document file://iam_policy.json

Criar IAM Service Account para o LBC

Service account no kube-system vinculado à role do Load Balancer Controller

$ eksctl create iamserviceaccount \
  --cluster=devops-matta-cluster \
  --namespace=kube-system \
  --name=aws-load-balancer-controller \
  --role-name AmazonEKSLoadBalancerControllerRole \
  --attach-policy-arn=arn:aws:iam::<ACCOUNT_ID>:policy/AWSLoadBalancerControllerIAMPolicy \
  --approve

Instalar AWS Load Balancer Controller via Helm

O LBC fica rodando no kube-system aguardando — provisiona o ALB apenas quando o Ingress for aplicado na Etapa 7.

$ helm repo add eks https://aws.github.io/eks-charts && helm repo update eks
$ helm install aws-load-balancer-controller eks/aws-load-balancer-controller \
  -n kube-system \
  --set clusterName=devops-matta-cluster \
  --set serviceAccount.create=false \
  --set serviceAccount.name=aws-load-balancer-controller
$ kubectl get deployment -n kube-system aws-load-balancer-controller
NAME READY UP-TO-DATE AVAILABLE
aws-load-balancer-controller 2/2 2 2
5 IAM Service Account dos pods + build da imagem
⚠ Atenção: No comando de push para o ECR, substitua <alias> pelo alias único do seu repositório público — disponível no console do Amazon ECR após criar o repositório.

IAM Service Account — acesso a S3 e DynamoDB

Pods da aplicação acessam recursos de dados sem credenciais hardcoded via IRSA

$ eksctl create iamserviceaccount \
  --cluster=devops-matta-cluster \
  --name=devops-matta-pod-role \
  --role-name DevOpsMattaPodExecutionRole \
  --attach-policy-arn=arn:aws:iam::aws:policy/AmazonS3FullAccess \
  --attach-policy-arn=arn:aws:iam::aws:policy/AmazonDynamoDBFullAccess \
  --region us-east-1 \
  --approve

Build e push da imagem para o ECR

Cria repositório público no ECR, builda e envia a imagem Docker

# Autenticar Docker no ECR público
$ aws ecr-public get-login-password --region us-east-1 | \
  docker login --username AWS --password-stdin public.ecr.aws
$ cd app-devops-matta/src
$ docker build -t app-devops-matta .
$ docker tag app-devops-matta:latest public.ecr.aws/<alias>/app-devops-matta:latest
$ docker push public.ecr.aws/<alias>/app-devops-matta:latest
6 Route 53 + ACM — domínio e certificado SSL
⚠ Atenção: O domínio app-devops-matta.com precisa estar registrado em um registrador antes desta etapa. Após criar a hosted zone, aponte os 4 name servers retornados no registrador. A propagação pode levar até 24h.

Criar a Hosted Zone no Route 53

O --caller-reference $(date +%s) gera um ID único automático para evitar criação duplicada da zona.

$ aws route53 create-hosted-zone \
  --name app-devops-matta.com \
  --caller-reference $(date +%s)
# Output: HostedZoneId + 4 name servers NS
# Exemplo de HostedZoneId: Z0481MXKF3GTR
# Consultar o HostedZoneId a qualquer momento
$ aws route53 list-hosted-zones \
  --query "HostedZones[?Name=='app-devops-matta.com.'].Id" \
  --output text

Solicitar certificado SSL wildcard no ACM

Anote o CertificateArn retornado — será usado na annotation do Ingress na Etapa 7.

$ aws acm request-certificate \
  --domain-name *.app-devops-matta.com \
  --subject-alternative-names app-devops-matta.com \
  --validation-method DNS \
  --region us-east-1
# Output: { "CertificateArn": "arn:aws:acm:us-east-1:
# <ACCOUNT_ID>:certificate/8F3B2A1C-4D5E-6F7G-8H9I-0J1K2L3M4N5O" }
# ⚠ Anote o CertificateArn — usado no Ingress YAML
📋 Próximo passo: Valide o certificado adicionando o registro CNAME gerado pelo ACM na hosted zone do Route 53. O certificado precisa estar com status ISSUED antes de aplicar o Ingress.

8 Registrar Alias records no Route 53 → ALB
ℹ Info: Dois IDs diferentes em jogo — Z0481MXKF3GTR é o HostedZoneId da sua zona DNS (gerado na Etapa 6). Z35SXDOTRQ7X7K é o HostedZoneId fixo do ALB na us-east-1, valor padrão da AWS para esta região.

Criar registros Alias A para Norte e Sul

Com o DNS do ALB obtido via kubectl get ingress, crie um Alias record para cada subdomínio.

# norte.app-devops-matta.com → ALB
$ aws route53 change-resource-record-sets \
  --hosted-zone-id Z0481MXKF3GTR \
  --change-batch '{
    "Changes": [{"Action": "CREATE", "ResourceRecordSet": {
      "Name": "norte.app-devops-matta.com", "Type": "A",
      "AliasTarget": { "HostedZoneId": "Z35SXDOTRQ7X7K",
        "DNSName": "k8s-frontend-xxxxxx.us-east-1.elb.amazonaws.com",
        "EvaluateTargetHealth": true }}}]}'
# sul.app-devops-matta.com → mesmo ALB
$ aws route53 change-resource-record-sets \
  --hosted-zone-id Z0481MXKF3GTR \
  --change-batch '{
    "Changes": [{"Action": "CREATE", "ResourceRecordSet": {
      "Name": "sul.app-devops-matta.com", "Type": "A",
      "AliasTarget": { "HostedZoneId": "Z35SXDOTRQ7X7K",
        "DNSName": "k8s-frontend-xxxxxx.us-east-1.elb.amazonaws.com",
        "EvaluateTargetHealth": true }}}]}'
# Testar acesso via HTTPS
$ curl -I https://norte.app-devops-matta.com
$ curl -I https://sul.app-devops-matta.com

Boas práticas aplicadas

🗂 Versionamento & IaC

Toda infra no Git via Terraform. State remoto no S3 com lock no DynamoDB. Nenhum recurso criado manualmente.

🔐 Segurança por padrão

IRSA via eksctl create iamserviceaccount. Zero credenciais nos pods. SSL automático via ACM. Least privilege por role.

📦 Isolamento por workload

Cada estado tem deployment, service, ConfigMap e recursos de dados (S3 + DynamoDB) completamente isolados.
